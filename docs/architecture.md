# Architecture Document - SPI Slave Verification Environment

## 1. Overview

This is a **non-UVM, class-based SystemVerilog** verification
environment. It follows the classic generator -> driver -> DUT ->
monitor -> scoreboard/coverage pipeline that almost every UVM
environment is also built on internally - just without the UVM base
class library, factory, or config_db. This makes the architecture
easy to read top-to-bottom and easy to explain in an interview.

## 4. Clock and Reset Strategy

- `i_Clk` (system clock, 100 MHz / 10ns period): generated in
  `top_tb.sv` with a simple `always #5 i_Clk = ~i_Clk;` free-running
  clock.
- `i_SPI_Clk` (SPI bus clock, 10 MHz / 100ns period): generated by the
  **driver**, not by `top_tb.sv`, since the driver is acting as the
  SPI master and the SPI clock is only toggling during an active
  transfer (idles low in Mode 0 between bursts).
- `i_Rst_L` (active-low reset): driven by `driver::reset_dut()`,
  called once at the start of every test via `environment::reset_env()`.
  The reset test (`test::reset_test()`) additionally re-asserts reset
  mid-transaction to check recovery behavior.

## 5. Why Not UVM?

This project intentionally avoids UVM, the factory pattern, RAL, and
advanced inheritance, per the project's design goals. The class-based
architecture above demonstrates the same core verification concepts
(transactions, generator/driver/monitor/scoreboard, mailboxes,
functional coverage, SVA) that UVM is built on, without the extra
framework overhead - appropriate for a fresher-level portfolio project
where the reviewer wants to see verification *fundamentals*, not
framework memorization.

## 6. Communication Mechanisms

| Link | Mechanism | Purpose |
|---|---|---|
| generator -> driver | `mailbox #(transaction) gen2drv_mbx` | Decouples stimulus generation from bus driving |
| monitor -> scoreboard | `mailbox #(transaction) mon2scb_mbx` | Passes observed bus transactions for checking |
| monitor -> coverage | `mailbox #(transaction) mon2cov_mbx` | Passes observed bus transactions for coverage sampling |
| driver -> scoreboard | direct handle (`scb.push_expected_miso()`) | Tells scoreboard what MISO byte to expect next |
